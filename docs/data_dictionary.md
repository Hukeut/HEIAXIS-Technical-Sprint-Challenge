# Data Dictionary & Synthetic Dataset Notes

**High Level Explanation:** This whole dataset is made up. There are no real students, no real records, nothing that could ever be traced back to an actual person. Every name, every number, every event was generated by a computer program following rules we wrote ourselves.

Here's why it exists. To prove that our early-warning idea actually works, we needed something that behaves like real school records, not just a pile of random numbers. Real schools keep information in a lot of different places: attendance is tracked one way, whether a student uses the online course system is tracked another way, how a student says they're feeling is a short separate survey, and when a counselor or advisor reaches out to a student is tracked somewhere else entirely. So instead of one simple spreadsheet, we built seven separate files that all talk about the same students, the same way a real school's different offices each keep their own records while describing the same person.

We also didn't make every fake student the same. About half are just fine, nothing going on. But we deliberately wrote in different kinds of stories: some students who slowly become less engaged over the weeks, some who reached out for help and whose case then got dropped by the school (a request that was opened and never closed, for example), some who had a rough start but recovered, and a few who look concerning at first glance but actually aren't declining at all, they're simply quiet by nature. That last group exists specifically to test whether our detection rules are smart enough not to unfairly flag someone just because they're naturally reserved rather than actually struggling.

On top of all that, we made the data messy on purpose: inconsistent spelling of office names, a few duplicate records, a handful of clearly wrong numbers, and students who don't always fill out the weekly survey. Real school data is never perfectly clean, so if our fake data were too tidy, it wouldn't prove much about whether the system can actually handle the real world.

Think of the whole thing like a flight simulator. Nobody actually crashes, and no real plane is ever at risk, but the controls behave the way real ones would. That's what lets the instruments get tested and proven before they're ever pointed at a real student.

**Low Level Explanation:** Five operational tables (students, weekly engagement, belonging pulse, care interactions, weekly outcomes) plus two reference tables added beyond the brief's stated minimum: an academic calendar and a staff roster. The calendar exists so a dip during a known high-load week, like midterms or a break, isn't scored the same as an unexplained dip elsewhere in the term. The staff roster exists so a stalled office can be read as a capacity problem (a high caseload per staff member) rather than implicitly read as one person's negligence, which matters given this data is eventually meant to sit behind a real, sensitive product. No demographic or identity attributes were modeled; they weren't necessary for the mechanics under test, and a synthetic proxy for a protected characteristic has no place near a risk-flagging exercise, even a fabricated one. The dataset is also deliberately messy: inconsistent office-name casing, a handful of duplicate interaction rows, a few out-of-range values, and realistic survey non-response, because a schema that's too clean wouldn't actually exercise the cleaning and validation step the prototype is meant to demonstrate. (Full reasoning and the alternatives considered are in `docs/product_interpretation_memo.md`, under Data Selection Rationale.)

*All data below is synthetic. No real student, staff, or institutional data was used or referenced at any point. Student IDs, staff IDs, and interaction IDs are arbitrary generated identifiers with no correspondence to any real system. Generated with `src/generate_data.py`, seeded (`random.seed(42)`) for reproducibility: 700 students, 7 weeks, 19 staff across 5 offices.*

## students.csv
| field | type | description |
|---|---|---|
| student_id | string | Synthetic identifier, e.g. `S0001` |
| program | string | One of 7 fabricated academic programs |
| class_year | string | Freshman / Sophomore / Junior / Senior |
| enrollment_status | string | `active` for all rows in this dataset (see Assumptions) |

## staff.csv
| field | type | description |
|---|---|---|
| staff_id | string | Synthetic identifier, e.g. `ST001` |
| office | string | One of 5 canonical offices (Counseling, Academic Advising, Financial Aid, Dean of Students, Residential Life) |
| role | string | Case Manager / Advisor / Coordinator / Director |

Office staffing is deliberately uneven (2-6 staff per office) so that per-office
caseload pressure is a real, discoverable pattern rather than a flat baseline.

## engagement_weekly.csv (one row per student per week)
| field | type | description |
|---|---|---|
| student_id | string | FK to students.csv |
| week_number | int | 1-7 |
| attendance_rate | float | 0.0-1.0 (raw data contains ~1% rows >1.0, injected on purpose, see cleaning) |
| lms_logins | int | Weekly LMS login count (raw data contains a handful of negative values, injected on purpose) |
| lms_activity_score | float | 0-100 composite activity proxy |
| sessions_missed | int | Missed class/advising sessions that week |
| participation_score | float | 0-10 |

## belonging_pulse.csv (one row per student per week)
| field | type | description |
|---|---|---|
| student_id | string | FK to students.csv |
| week_number | int | 1-7 |
| survey_submitted | bool | Whether the student responded to that week's pulse survey (~65% baseline response rate, intentionally realistic non-response) |
| belonging_score | float or blank | 1-5 self-reported belonging; blank when not submitted (never imputed) |
| peer_interaction_count | int or blank | Self-reported peer interactions that week; blank when not submitted |

## care_interactions.csv (one row per interaction event)
| field | type | description |
|---|---|---|
| interaction_id | string | Synthetic identifier, e.g. `C00001` |
| student_id | string | FK to students.csv |
| date | date | Interaction date within the synthetic term |
| office | string | Raw office name: **deliberately inconsistent casing/abbreviation** across rows (e.g. "Counseling", "counseling", "Counseling Center") to simulate multi-source ingestion; normalized during cleaning |
| interaction_type | string | outreach / referral / warm_handoff / staff_note |
| response_status | string | responded / no_response / pending / n_a (n_a for types where it doesn't apply) |
| referral_status | string | open / closed / n_a |
| handoff_owner | string or blank | staff_id responsible for a warm_handoff; **blank is meaningful** (it means nobody is on record as owning the handoff) |
| note_category | string | academic / wellbeing / financial / behavioral / social |

This table also contains ~1.5% duplicate rows (same interaction logged twice) to
simulate a realistic multi-office ETL artifact, removed during cleaning.

## weekly_outcome.csv (one row per student per week)
| field | type | description |
|---|---|---|
| student_id | string | FK to students.csv |
| week_number | int | 1-7 |
| status | string | persisted / disengaged / referred / unresolved / improved / dropped_course / no_action_needed |

This field is **generated alongside the rest of the synthetic data, not predicted
by the prototype.** It represents a plausible downstream field an institution would
already track (e.g. from a case management system), included so the schema is
realistic, not as a label the detectors are scored against.

## academic_calendar.csv (reference table, not in the original brief)
| field | type | description |
|---|---|---|
| week_number | int | 1-7 |
| week_label | string | e.g. `midterms`, `fall_break`, `regular`; context for interpreting dips |

Added so a legitimate dip during a known high-load or low-attendance week (midterms,
a break) isn't treated identically to an unexplained dip. Not currently used to
automatically suppress flags; it's available as context for a human reviewer, not
as an automatic filter (see `docs/evaluation_logic.md`).

## _ground_truth_archetypes.csv (generator-only, not an operational table)
| field | type | description |
|---|---|---|
| student_id | string | FK to students.csv |
| archetype | string | The hidden pattern used to generate that student's rows: `stable`, `disconnecting`, `care_gap`, `improving`, `dropped_course`, `noisy_false_flag_bait` |

**A real prototype would never have access to this file.** It exists only to support
`src/self_consistency_check.py`, and is explicitly excluded from anything the
detection logic reads. See `docs/evaluation_logic.md` for why this is not validation.

## How the data was generated

`src/generate_data.py` assigns each student one hidden archetype (weighted random
draw), then generates all of that student's rows across every table so the pattern
is consistent within a student but variable across students:

- **stable (52%):** normal week-to-week noise, no directional drift.
- **disconnecting (16%):** steady decline across attendance, LMS activity,
  participation, and belonging, plus declining survey response rate.
- **care_gap (13%):** engagement drifts only mildly (the student's behavior is
  *not* the primary signal); instead, care_interactions rows are biased toward
  open referrals with no closure, unanswered outreach, and unowned handoffs.
- **improving (8%):** early dip, recovers by week 5-7 (tests that the detector
  doesn't keep flagging students who've already recovered).
- **dropped_course (5%):** sharp engagement collapse starting week 4.
- **noisy_false_flag_bait (6%):** low *absolute* engagement and belonging scores
  from week 1 onward, but flat, no decline. Exists specifically to test whether
  the detection logic wrongly flags low baseline values instead of genuine change.

Data-quality noise is injected on top of all archetypes (not tied to any one
pattern): ~1% of attendance values pushed above 1.0, a handful of negative login
counts, inconsistent office-name casing throughout, and ~1.5% duplicate
interaction rows.

## Assumptions

- A single synthetic academic term, 7 weeks, starting 2026-02-02 (arbitrary).
- `enrollment_status` is `active` for all students in this dataset; withdrawal
  or leave-of-absence was not modeled as a distinct status, since `dropped_course`
  already exercises the "student outcome changes mid-term" case without adding a
  second, overlapping status dimension.
- Pulse survey response (~65% baseline) is treated as its own signal, not
  imputed; a student who stops responding to surveys is informative in a
  different way than a student who responds and reports low belonging.
- Office names in `care_interactions.csv` are intentionally inconsistent to
  simulate multi-source origin (e.g. different case-management systems per
  office); the canonical mapping used during cleaning is in `src/cleaning.py`.
- `weekly_outcome.status` is a plausible downstream institutional field, not a
  model target (see note above).

## Additional Metrics Considered for a Larger-Scale System

Not modeled here, either because of the sprint's time budget or because they raise questions this prototype isn't equipped to answer responsibly. Split into two groups for that reason: some of these would likely get added as the product matures, others would need real governance work before they should ever be added at all.

**Left out mainly for time and scope. Reasonable next additions at scale.**

- **Prior-term or multi-semester baseline data.** This prototype compares a student to themselves within a single 7-week term, using weeks 1-2 as the baseline. A real system with a student's history across multiple terms could build a much sturdier personal baseline and catch slower, multi-term declines this design can't see. Left out because a single-term synthetic dataset was enough to prove the mechanic, and modeling multiple terms would have meant generating and explaining several times more data for the same time budget.
- **Academic performance signals: grades, assignment submission timeliness, missed deadlines.** Deliberately left out of this version, partly for time, but also because final grades tend to arrive too late to count as an "early" signal, by the point a grade reflects a problem, it's often already visible elsewhere. Interim signals like missing assignments or late submissions would be a genuinely useful addition at scale, closer in spirit to attendance and LMS activity than to a final grade.
- **A case audit and escalation event log**, timestamping every touch on a case (opened, note added, reassigned, closed), rather than the single-snapshot interaction rows used here. This is exactly the kind of table a real auditability requirement (see `docs/architecture.md`) would need. Left out because it meaningfully increases generation complexity for a detail that doesn't change what the current detectors can prove.
- **An identifier crosswalk across office systems.** Real institutions run separate systems per office with inconsistent student identifiers; this prototype assumes one clean `student_id` throughout. Left out because building and explaining realistic ID mismatches would have consumed schema-design time without adding new detection logic to show.
- **Financial and basic-needs indicators**: financial holds, work-study hours, meal plan usage trends. Financial precarity is a well-documented driver of disengagement and would plausibly strengthen the student-side signal. Left out here both for time and because it's a data category the brief didn't ask for, but at scale it's a strong candidate, with real financial-data handling safeguards attached.
- **Lightweight structured tags from free-text staff notes**, instead of the single `note_category` field used here. Richer than a category label, short of full NLP. Left out to avoid adding a text-processing dependency to a prototype that was supposed to stay simple and explainable.

**Left out because they raise real privacy and ethical questions, not just a time constraint. These need governance work before they belong in the product at all.**

- **Health center or disability services visit records.** Even more sensitive than the wellbeing data already modeled, and touches protected health information rather than general educational records. Adding this would need its own legal and ethical review, not just an engineering decision.
- **Physical presence data**, campus ID card swipes, building access logs, wifi login patterns. Technically a strong engagement proxy, but it's also passive location tracking of students who never opted into being monitored that way. This is the kind of metric that should be argued against by default, not added just because the data exists somewhere.
- **Peer social graph data**, who a student actually interacts with, not just how often they say they do. More informative than the self-reported `peer_interaction_count` used here, but it necessarily includes data about other students who didn't consent to being part of someone else's case file. Meaningfully different, and riskier, than data about one student's own behavior.
- **Sentiment or content analysis of student communications** (emails, chat messages with advisors). The most invasive item on this list: reading what a student actually wrote, rather than whether an interaction happened. Even with good intentions, this is close to surveillance, and it's exactly the kind of "black box conclusion from messy signals" the brief explicitly warns against.
